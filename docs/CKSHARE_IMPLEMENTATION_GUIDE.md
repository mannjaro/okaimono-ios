# CKShareで買い物リストを共有する手順（初心者向け）

この文書は、あなたが自分で実装するための**進め方メモ**です。完成コードのコピペではなく、「何を、なぜ、どの順で」やるかを示します。

---

## この機能で目指すこと

家族や友人に、**買い物リスト1件**を共有できるようにします。

共有すると、そのリストの中身もまとめて相手に見えます。

```text
ShoppingList（買い物リスト） ← これを共有する
 ├── MenuItem（献立）
 │    └── Ingredient（材料）
 └── MenuItem
      └── Ingredient
```

相手は献立や材料の追加・編集、購入チェックもできます（共同編集）。

---

## 今のアプリと、これから増えるもの

### 今できていること

- Core Dataでリスト・献立・材料を保存している
- 同じ人の別デバイス間では、CloudKitで同期される想定になっている
- データを置く場所（ストア）は**1つ**だけ

### これから必要なこと

他の人のデータも受け取る場所が要るので、ストアを**2つ**にします。

| 名前 | たとえ | 何が入るか |
|------|--------|------------|
| privateストア | 自分の引き出し | 自分が作ったリスト |
| sharedストア | 共有フォルダのコピー | 他の人から共有されたリスト |
| CKShare | 「このフォルダを誰とシェアするか」の設定 | 招待・権限の情報 |

```text
画面（SwiftUI）
    │
    ▼
viewContext（データの読み書き窓口）
    │
    ├── privateストア  ←→  CloudKit の private データベース（自分用）
    └── sharedストア   ←→  CloudKit の shared データベース（共有用）
```

画面から見ると、どちらのストアのデータも同じリスト一覧に並びます。裏で置き場所が違うだけです。

---

## 用語の超短い説明

読みながら分からなくなったらここに戻ってください。

- **Core Data** … アプリ内のデータベース（リストや材料を保存する仕組み）
- **ストア / SQLite** … 実際にファイルとして保存されている場所。今は1ファイル。共有対応後は2ファイル
- **CloudKit** … Appleのクラウド。iCloud経由で同期・共有する
- **CKShare** … 「このデータを誰とシェアするか」を表すCloudKitの仕組み
- **オーナー** … リストを最初に作って共有した人
- **参加者** … 招待を受け取って一緒に使う人
- **persistent store description** … 「どのファイルを」「どのCloudKitに繋ぐか」を決める設定オブジェクト
- **assign** … 新しいデータを、どちらのストアに入れるか指定する操作
- **SceneDelegate / AppDelegate** … アプリ起動や「共有リンクをタップした」など、システムのイベントを受け取る入口。SwiftUIだけでは足りない場合がある

## Step 1: データを置く場所を2つにする（最重要）

対象ファイル: `okaimono-app/App/Persistence.swift`

### なぜ必要？

今は「自分の引き出し」しかありません。  
他の人から共有されたデータ用の引き出しがないと、招待を受け取れません。

### イメージ

```text
【今】
okaimono_app.sqlite  ← 全部ここ

【あと】
okaimono_app.sqlite         ← 自分のリスト（private）※場所はそのまま
okaimono_app-shared.sqlite  ← 共有されたリスト（shared）※新規
```

### やること（順番）

1. 今あるストア設定を「private用」と明示する  
   - CloudKitの `databaseScope` を `.private` にする
2. shared用の設定をもう1つ作る  
   - **ファイル名は別**にする（例: `okaimono_app-shared.sqlite`）
   - CloudKitの `databaseScope` を `.shared` にする
   - 使うCloudKitコンテナIDは今と同じ（`iCloud.mannjaro.okaimono-app`）
3. 両方に次を付ける（今private側にある設定と同じもの）
   - 自動マイグレーション
   - Persistent History Tracking
   - Remote Change Notification
4. コンテナに「ストアは2つある」と教える
5. ロード完了の判定を直す  
   - `loadPersistentStores` は**ストアごとに1回**呼ばれる  
   - **2回とも成功してから** `isStoreLoaded = true` にする  
   - `privatePersistentStore` と `sharedPersistentStore` を別々に覚える
6. `resetLocalStore()` も2ファイル対応に直す  
   - 今は「1ファイルだけ消す」前提になっている

### このステップでやらないこと

- 共有ボタンはまだ作らない
- CKShareのコードはまだ書かない
- 招待の受け入れはまだ書かない

### 完了チェック

- [ ] アプリが起動する
- [ ] 今まで通り、リストの作成・編集・削除ができる
- [ ] 既存のリストが消えていない
- [ ] テストが通る（CloudKitをオフにしたローカルテスト含む）

うまくいったら、**「自分用と共有用の引き出しができた」**状態です。中身の共有はまだです。

## Step 2: 新しいデータを「正しい引き出し」に入れる

### なぜ必要？

引き出しが2つになると、Core Dataは「新しい献立をどっちに入れる？」で迷います。  
特に、**共有されたリストに献立を足すとき**、別の引き出しへ入れて親と結びつけようとすると保存エラーになります。

### ルール（これだけ覚える）

1. **新しい買い物リスト** → いつも **private**（自分が作るもの）
2. **新しい献立・材料** → **親と同じストア**  
   - 共有リストの子供 → shared  
   - 自分だけのリストの子供 → private

### やること

1. `ShoppingListView` でリストを作るとき  
   - 作成 → privateストアへ `assign` → 名前設定 → 保存
2. `MenuItemList` で献立を作るとき  
   - 作成 → **親のShoppingListと同じストア**へ `assign` → 親と結びつける → 保存
3. `IngredientListContent` で材料を作るとき  
   - 作成 → **親のMenuItemと同じストア**へ `assign` → 親と結びつける → 保存
4. 同じ処理が増えるので、小さな共通メソッド（例: 「親と同じストアへ入れる」）にまとめる

親がどのストアにいるかは、だいたい次の考え方です。

```text
親オブジェクト
  → objectID
  → persistentStoreCoordinator に「このIDはどのストア？」と聞く
  → そのストアへ新しい子を assign
```

### このステップでやらないこと

- 共有UIはまだ作らない

### 完了チェック

- [ ] 新しい自分のリストが作れる
- [ ] 献立・材料の追加が今まで通り動く
- [ ] この段階ではまだ「共有されたリスト」はないので、自分のデータが壊れないことを優先確認

## Step 3: 「共有の操作」を1か所にまとめる

新しいファイルの候補: `okaimono-app/App/CloudSharingService.swift`  
（名前は何でもよい。画面からCloudKitを直接叩かないための箱）

### なぜ必要？

画面にCloudKitの細かい処理を書くと、すぐ読みにくくなります。  
「共有する」「既にある共有を取る」「権限を見る」を1つの場所にまとめます。

### 最初に用意したい操作

| 操作 | 意味 |
|------|------|
| 共有を作る | リストに対して `CKShare` を新規作成 |
| 共有を取得 | すでに共有済みなら、その `CKShare` を取る |
| 招待を受け入れる | 共有リンクから来た情報を sharedストアへ入れる |
| 更新できる？ | 編集UIを出すかどうかに使う |
| 削除できる？ | 削除ボタンを出すかどうかに使う |

### 覚えどころ

- 新しい共有: `container.share([list], to: nil)`  
  - `nil` = 「新しい共有を作って」という意味合い
- 既存の共有を取る: `container.fetchShares(matching: ...)`
- 共有タイトルにリスト名を入れると、相手側の共有画面で分かりやすい

### 完了チェック

- [ ] サービスを作った
- [ ] まだ画面にボタンがなくてもよい（次のステップでつなぐ）
- [ ] 「リスト1件を渡すと共有できる」形になっている

## Step 4: 共有ボタンとシステム共有画面をつなぐ

### なぜ必要？

招待メールやメッセージの送信、参加者の管理は、自分でゼロから作らなくてよいです。  
Appleの標準画面（`UICloudSharingController`）に任せます。

### どこにボタンを置く？

いちばん分かりやすい場所は **リスト詳細（`DetailView`）の右上** です。  
ここでは「どのリストを共有するか」がすでに決まっています。

表示の目安:

| 状態 | ボタンのイメージ |
|------|------------------|
| まだ共有していない | 「共有を開始」 |
| 自分がオーナー | 「共有を管理」 |
| 自分が参加者 | 「共有情報」 |
| 読み込み中 | くるくる（ProgressView） |

### SwiftUIでの注意

`UICloudSharingController` は UIKit の画面なので、`UIViewControllerRepresentable` で包んで sheet から出す形になります。  
（「SwiftUIから古いUIKit画面を借りる」イメージ）

### 共有の方針（今回決めたこと）

- 誰でも入れる公開リンクではなく、**招待制**
- 相手は **読む＋編集できる**（read-write）

### 完了チェック

- [ ] 詳細画面から共有画面が開く
- [ ] 共有のタイトルにリスト名が出る
- [ ] まだ相手側の受け入れまで行かなくてよい（次のStep）

まずは `DetailView` だけで十分です。一覧の共有アイコンは、このあとで足しても構いません。

## Step 5: 招待リンクを開けるようにする

ここが「SwiftUIだけでは足りない」部分です。

### なぜ必要？

相手がメッセージのリンクをタップすると、システムがアプリに「この共有を受け入れて」と渡します。  
その受け取り口が、**SceneDelegate / AppDelegate** です。

### やること

1. `Info.plist` に次を追加する（これが無いと共有リンクでアプリが開けない）

```xml
<key>CKSharingSupported</key>
<true/>
```

2. `SceneDelegate` を作る  
   - アプリ起動中にリンクを開いたとき  
   - アプリが閉じている状態からリンクを開いたとき  
   の両方で、共有情報（`CKShare.Metadata`）を受け取る
3. `AppDelegate` を作る  
   - 「新しい画面（scene）には、この SceneDelegate を使って」と指定する
4. `okaimonoApp.swift` で `@UIApplicationDelegateAdaptor` を使い、AppDelegate をつなぐ
5. 受け取った情報を、**必ず sharedストア指定**で受け入れる

```text
metadata を受け取る
  → sharedPersistentStore の準備ができているか確認
  → できていれば acceptShareInvitations
  → まだなら「後で処理する箱」に入れておく
  → 両ストアのロードが終わったら処理する
```

### よくあるつまずき

起動直後はストアがまだ読み込み中です。  
「すぐ accept」すると失敗しやすいので、**待ってから受け入れる**設計にしてください。

### 完了チェック

- [ ] Info.plist に `CKSharingSupported` がある
- [ ] 実機Aから招待 → 実機Bでリンクを開ける
- [ ] Bの一覧に、共有されたリストが表示される（少し待つ必要がある場合あり）

## Step 6: 共同編集が安全に動くか確認する

### うれしい話

sharedストアも同じ `viewContext` につながっているので、  
今の `ShoppingListView` の `@FetchRequest` は、共有リストもまとめて一覧に出せます。  
「共有専用リスト画面」をゼロから作る必要はありません。

### やること

1. 参加者が、共有リストに献立・材料を追加できることを確認する（Step 2の assign が効く場面）
2. 編集・削除の前に、権限を確認する  
   - `canUpdateRecord` … 編集していいか  
   - `canDeleteRecord` … 削除していいか
3. 権限が無いときはボタンや入力を消す／無効にする  
   - 今回の仕様は基本 read-write だが、同期中や設定変更に備えて入れておくと安心

触る画面の目安:

- `ShoppingListView`
- `MenuItemList`
- `ArchiveMenuList`
- `IngredientListContent`
- `CartView`

### 完了チェック

- [ ] Aが追加したものが、しばらくするとBに見える
- [ ] Bが追加したものが、しばらくするとAに見える
- [ ] カートのチェックも双方で同期される

**注意:** CloudKitはチャットのようなリアルタイムではありません。  
数秒〜場合によってはもう少し待つことがあります。「壊れた」と即断しないでください。

## Step 7: 「消す」の意味が3種類あることをUIで分ける

次の3つは別物です。最初から全部混ぜないこと。

| 操作 | 誰がやる | 結果のイメージ |
|------|----------|----------------|
| リストを削除 | 主にオーナー | 献立・材料も消える。参加者側からも消える可能性が高い |
| 共有を停止 | オーナー | 相手はもう見られなくなる。自分のリストを残すかは別問題 |
| 共有から退出 | 参加者 | 自分の画面からその共有リストが消える |

### やること

1. 共有中のリストを削除するときは、確認ダイアログを出す
2. 「共有停止・退出」は、まず標準の共有管理画面に任せる
3. 「共有を止めたあともオーナーの手元にコピーを残す」は難易度が高いので、**最初の版では後回しでよい**

### 完了チェック

- [ ] ユーザーが「削除」と「退出」を間違えにくい
- [ ] 共有中にいきなり消える危険な操作に確認がある

## Step 8: 画面が更新されないときの対処（必要なら）

今のアプリにはすでに次があります。

- Remote Change Notification（クラウドから変化が来た合図）
- `automaticallyMergesChangesFromParent = true`（変化を画面用データに取り込む）

### まずは試す

招待後や相手の編集後、一覧が自然に更新されるか見る。

### 更新されないときだけ追加する

- `.NSPersistentStoreRemoteChange` を監視する
- 履歴（Persistent History）を読み、`viewContext` に反映する

最初から完璧に作らなくて大丈夫です。「必要になったら」で十分です。

## Step 9: 材料名のサジェスト範囲を決める

`IngredientListContent` は、今は材料をかなり広く検索して候補を出しています。

共有を入れると、**他の人が作った材料名まで候補に出る**可能性があります。

最初の実装ではどちらかに決めてください。

- **おすすめ（安全）:** 今開いている買い物リストの中だけから候補を出す
- **広い方:** 自分が見られる全リスト（共有含む）から候補を出す

正解はありませんが、迷ったら「今のリストの中だけ」から始めると分かりやすいです。

## Step 10: テスト

### 自動テストで確認しやすいこと

`okaimonoAppTests` 向け:

- ストアが2つロードされる
- 両方が終わる前に「読み込み完了」にしない
- 新しいリストは private に入る
- 親と同じストアに子が入る
- ストア準備前の招待情報を保留できる

実CloudKitの招待そのものは、自動テストでは再現しにくいです。ここは実機が本命です。

### 実機チェックリスト（2台）

1. Aでリスト・献立・材料を作る
2. AからBを招待する
3. Bがアプリ起動中にリンクを開く
4. Bがアプリ終了状態からリンクを開く
5. Bが献立を追加する
6. Aに反映されるまで待つ
7. 双方でカートのチェックを変える
8. 一度オフラインで編集し、復帰後に同期されるか見る
9. Aが共有を管理する
10. Bが退出する

## Step 11: TestFlight / 本番向け

開発（Development）で動いてから、次をやります。

1. CloudKit Console でスキーマを Production にデプロイする
2. Archiveして entitlements（CloudKitなど）を確認する

## おすすめの進め方（短い版）

```text
1. 2ストア化           ← 土台。ここが一番大事
2. 保存先の指定         ← 共同編集で壊れないための保険
```

各段階で「ビルド → 既存機能確認 → 次へ」を繰り返してください。

## 困ったときのよくある原因

| 症状 | 疑うポイント |
|------|----------------|
| 招待リンクを開いてもアプリが共有を受け取らない | `CKSharingSupported`、SceneDelegate接続 |
| 受け入れ直後にエラー | sharedストア未ロードのまま accept している |
| 共有リストに献立を足せない | 子を親と同じストアへ `assign` していない |
| 相手にデータが来ない | 待ててない／ネットワーク／iCloudサインイン／CloudKit設定 |
| TestFlightだけ失敗 | Productionスキーマ未デプロイ |
